### 2026-04-26 — Tried to reproduce search hang verbatim before treating it as perf

**What happened:** User reported "list is loading indefinitely" after typing in iOS/iPadOS search. I built+launched the app on both iPhone and iPad, opened search, typed "100m offers", and saw the result come back in <1s on a 725-file vault. I cycled through fresh launches, iPad vs iPhone, different queries ("ios"), and different sims trying to reproduce the literal "indefinite" symptom before pivoting to "the perf is bad enough on a real device/vault that the user perceives it as indefinite, even if my sim is fast enough that it isn't."

**Why this was wrong:** With a benchmark showing 90ms cold to read 7.7MB of frontmatter for 725 files on the host, the underlying perf bug (`IOSNoteSearchSheet` reads frontmatter for every note just to filter by name) was obvious from a code read. I should have spent 5-10 minutes reproducing, then pivoted to "the symptom won't reproduce on a small fast vault, but the code is clearly doing N file reads it doesn't need" and gone straight to the fix.

**What better looks like:** When the user's symptom is clearly perceptual (slow → indefinite, choppy → broken), don't keep trying to reproduce the exact phrasing. Read the code path, find the hot path that justifies the perception, fix that, and move on. Reproduction is a sanity check for the FIX (verify it's now fast), not a gate to start.

### 2026-04-26 — Bug 010 fix introduced Bug 011 because I didn't trace the ID downstream

**What happened:** Bug 010 fix switched the search loader to `.fileOnly` for speed. That made `row.noteID = nil` and the search result's `note.id` fell back to `VaultDirectoryLoader.stableID(for: url)` (a path hash). I didn't notice that the very next thing the tap path does is call `MarkdownNoteStore.note(withID:)`, which walks the entire vault reading the FULL content of every `.md` file searching for a matching frontmatter UUID. With a non-matching path-hash, the lookup is forced into its absolute worst case (read every file, return nil) on every search-result tap. On a real device with iCloud-backed files this is enough to trip the iOS watchdog and crash the app.

**Why this was wrong:** I optimized the LIST load without auditing what consumed the resulting note's identity. Two distinct code paths (search load + tap-open) shared a value (note.id) and I changed one in a way that broke the other. Single-file mental model on a graph problem.

**What better looks like:** When changing how an entity's identity is derived (UUID vs path-hash vs whatever), grep for every consumer of that field BEFORE shipping the change. Especially fields named `id` or `noteID`. The scan is mechanical: `git grep -n "\.id\b" Noto/` for the entity, follow each call site, and verify the change of derivation doesn't silently break invariants downstream. Cost: 10 minutes. Reward: not shipping a regression that crashes the app on real devices.
