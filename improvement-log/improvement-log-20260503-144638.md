## Log

### 2026-05-03 — Spent ~10 minutes trying to tap a SwiftUI ToolbarItem back button by point

**What happened:** While reproducing/verifying bug 014, I needed to tap the editor's history-back button (a `ToolbarItemGroup(placement: .navigationBarTrailing)` button). The button has an accessibility identifier (`history_back_button`) and label (`Back`), but neither `flowdeck ui simulator tap "history_back_button" --by-id` nor `tap "Back"` (label) found it — and `flowdeck ui simulator find` returned `not_found`. I then tried tapping by point at (944,55), (922,55), (920,60), (975,55) — none registered. The pre-fix tap at (944,55) had appeared to work earlier, which sent me on a wrong-coordinate hunt.

**Why this was wrong:** I burned time trying to brute-force the coordinates. FlowDeck has `flowdeck ui simulator back` as a first-class command for back navigation; one call worked instantly.

**What better looks like:** When a button isn't surfaced in the accessibility tree (`find` returns not_found, no label/id match), don't fall through to point-tapping by guessing coordinates. Check `flowdeck ui simulator --help` or `--examples` for high-level navigation commands first (`back`, `key`, `swipe` from edge). For SwiftUI toolbar items specifically, the system nav bar often hides items from the in-process accessibility query — `flowdeck ui simulator back` is the right escape hatch.
