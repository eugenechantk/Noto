## Log

### 2026-03-28 — Route-based navigation lost isNew flag
**What happened:** When refactoring from boolean navigation flags to NoteRoute enum + navigationDestination(for:), the `isNew: true` parameter was not carried through the route. New notes no longer auto-focused the editor.
**Why this was wrong:** The old `navigationDestination(isPresented: $shouldNavigateToNew)` explicitly passed `isNew: true`. The new route enum defaulted to `false`.
**What better looks like:** When refactoring navigation patterns, audit every parameter that the old pattern passed to destination views. The Maestro E2E test caught this before a user would have.

### 2026-03-28 — Route-based navigation creates new store instances
**What happened:** The `navigationDestination(for:)` handler creates a new `MarkdownNoteStore` from the route's URLs. When the editor's `onDisappear` saves to this new store, the root store's items array doesn't update.
**Fix applied:** Added `.onAppear { store.loadItems() }` to FolderContentView so it refreshes from disk when returning from pushed views.
**What better looks like:** When passing model objects through Hashable routes, be aware that you lose reference identity. Either reload on appear or find a way to share the store instance.

### 2026-03-28 — Maestro + iOS 26 keyboard interaction
**What happened:** On iOS 26, Maestro's `hideKeyboard`, swipe-from-edge gestures, and status bar taps all dismiss the entire app when the keyboard is showing. This is likely due to iOS 26 Liquid Glass gesture handling changes.
**Workaround:** Use `stopApp` + `launchApp` (without clearState) to test persistence instead of navigating back in-app. This is actually a stronger test since it verifies disk persistence.
**Status:** Open -- need to revisit when Maestro updates for iOS 26 compatibility.

### 2026-03-28 — Stopped working mid-task waiting for user input when screenshots loaded
**What happened:** Multiple times during the Maestro debugging loop, I would read a screenshot, see the result, and then stop — waiting for the user to respond instead of immediately continuing to analyze and fix. Eugene had to say "why did you stop?" and "you should keep going" twice.
**Why this was wrong:** The task was clear: run E2E tests and fix problems. When a test fails and I see the screenshot, the next step is obvious — analyze the failure and fix it. There was no decision fork requiring user input. I should have kept moving.
**What better looks like:** When in an iterative debug loop (run test → check screenshot → diagnose → fix → rerun), never pause for user input unless there's a genuine fork (e.g., "should we change the app design or change the test?"). Screenshots are diagnostic data, not conversation prompts. Read them, draw conclusions, act immediately.

### 2026-03-28 — Maestro pressKey: Enter unreliable with UITextView
**What happened:** `pressKey: Enter` reports COMPLETED but doesn't create a newline in the UITextView. Content typed after Enter doesn't appear. Only the first `inputText` call works reliably.
**Workaround:** Tests focus on first-line (heading) content, which works. Multi-line editing is covered by unit tests instead.
**Status:** Open -- may be a TextKit 1 + Maestro interaction issue.
