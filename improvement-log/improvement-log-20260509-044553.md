## Log

### 2026-05-09 — Each NoteEditorSession spins up its own VaultController/rootStore

**What happened:** While debugging the macOS launch hang, found that `NoteEditorSession.init` defaults `vaultController = vaultController ?? VaultController(...)` — so when the parent doesn't pass one (which is true everywhere `NoteEditorScreen` is used today), every session creates a fresh `VaultController`, which in turn creates a fresh `MarkdownNoteStore` with its own copy of the vault item list. Multiple editor windows = multiple redundant scans. Fixed the immediate launch hang by making the scan run in the background, but architecturally the duplication is wasteful.
**Why this matters:** N redundant scans waste I/O, memory, and (on iCloud) coordinator handshakes. They also mean any cross-window operation that should observe a single source of truth has to reconcile across instances.
**What better looks like:** Hoist VaultController to the workspace level (alongside the workspace's MarkdownNoteStore) and pass a single shared instance down through `NoteEditorScreen` → `NoteEditorSession`. Treat `MainAppView`'s store and `VaultController.rootStore` as the same thing. Out of scope for the launch-hang fix, but should be tracked.
