### 2026-05-07 — Search sheet beachball: main-thread file reads via `MarkdownNoteStore.note(atVaultRelativePath:)`

**What happened:** User reported macOS beachball when opening the search sheet and when toggling scope. Diagnosed `NoteListView.swift` `appResult` and `loadRecentNotes` calling `rootStore.note(atVaultRelativePath:)` on the main actor — that path goes to `NoteRepository.noteRecord(at:)` which reads the entire file via `fileSystem.readString` to derive the title. With up to 60 search results × full file reads per scope toggle, the main thread spent seconds on I/O.
**Why this matters:** The SQLite search index already carries `noteID`, `fileURL`, `title`, `updatedAt` — re-reading each file to construct a `MarkdownNote` for a list row is pure waste. For recent notes the read can move into the existing detached task instead of running on main.
**What better looks like:** When wrapping search results into UI rows, never re-read the file. Trust the index. For any list-row construction that needs file metadata, do the I/O off main and keep main-actor work limited to fast struct init / `FileManager.fileExists`.
